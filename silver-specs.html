Before Java Silver

コードを書くだけでは見えにくい、Javaの仕様を先に読む。

Java Silverの問題は、文法を書けるだけでは止まります。どの修飾子が何を禁止するのか、どの型がどこまで約束するのか。ここでは、問題で迷いやすい仕様を短く整理します。

Tracea Guide

「覚える」より先に、「何を守るための仕様か」を見ましょう。

final、abstract、interface、privateは、単語だけ見ると硬いです。でも多くは「勝手に変えない」「未完成をそのまま使わない」「外から雑に触らせない」ためのルールです。

Silver問題では、出力よりも先に「これはコンパイルできるか」「このメソッドは上書きできるか」「この値は変わるか」を聞かれます。仕様の理由まで軽くつかんでおくと、選択肢を消す速度が上がります。

Lesson 01 / final

finalは「ここから先は変えない」というロック。

finalは場所によって意味が少し変わります。変数なら再代入できない。メソッドならオーバーライドできない。クラスなら継承できない。共通しているのは、後から勝手に変えられる範囲を狭くすることです。

final int max = 10;
// max = 20; // 再代入できない

final class Tool {
}
// class MyTool extends Tool {} // 継承できない
variable

final変数は再代入できない

値そのものを守るというより、変数が別の値を指すことを止めます。

method

finalメソッドは上書き不可

子クラスで動きを変えられたくない処理に使います。

class

finalクラスは継承不可

設計をこれ以上広げない、と宣言する形です。

Tracea

勘違い注意: final参照でも、中身が必ず不変とは限りません。

final List<String> list は、listに別のListを代入できないだけです。Listの中へaddできるかどうかは、また別の話です。Silverではここがよく引っかかります。

Lesson 02 / access modifier

アクセス修飾子は「誰が触れるか」を決める。

private、何も書かない、protected、publicの違いは、見える範囲の違いです。出力問題というより、コンパイルできるかを判定する問題で効きます。

public class Student {
  private int score;

  public int getScore() {
    return score;
  }
}
private

同じクラス内だけ

一番狭い。外から直接触らせたくない値に使います。

default

同じパッケージ内

何も書かない状態です。publicではありません。

protected

同じパッケージ + 継承先

privateより広いですが、publicほど全開ではありません。

public

どこからでも見える

公開APIとして使わせるものに付けます。

Tracea

勘違い注意: top-level classにprivate/protectedは付けられません。

普通のファイル直下のclassは、publicか何も書かない形です。private classやprotected classを見たら、まずコンパイルエラー候補として見てください。

前へ次のLessonへ

Lesson 03 / inheritance

継承は「共通部分を受け継ぎ、必要なら差し替える」仕組み。

extendsで親クラスのフィールドやメソッドを使えるようになります。ただし、何でも自由に触れるわけではありません。アクセス修飾子、コンストラクタ、オーバーライドの条件を一緒に見ます。

class Animal {
  void speak() {
    System.out.println("...");
  }
}

class Dog extends Animal {
  @Override
  void speak() {
    System.out.println("bow");
  }
}
extends

親の機能を受け継ぐ

共通処理を親に置き、子で使い回せます。

override

必要なら動きを変える

同じシグネチャのメソッドを子で書き直します。

super

親側を見る合図

親のコンストラクタやメソッドへ進むときに使います。

Tracea

勘違い注意: コンストラクタは継承されません。

子クラスのコンストラクタから親コンストラクタを呼ぶことはありますが、親のコンストラクタそのものが子へ受け継がれるわけではありません。

前へ次のLessonへ

Lesson 04 / abstract

abstractは「まだ完成していない部分がある」という印。

abstract classは直接newできません。abstract methodは中身を持たず、子クラスが実装する約束になります。未完成のまま使わせないための仕様です。

abstract class Shape {
  abstract int area();
}

class Square extends Shape {
  int area() {
    return 9;
  }
}
no new

abstract classは直接new不可

設計の途中なので、そのまま実体にできません。

no body

abstract methodは本体なし

波かっこではなく、セミコロンで終わります。

implement

子クラスが完成させる

未実装のメソッドを子が実装します。

Tracea

勘違い注意: abstractとfinalは同時に成立しません。

abstractは「子に完成させてほしい」、finalは「子を作らせない」。方向が逆なので、同じクラスに両方付けるとコンパイルエラーです。

前へ次のLessonへ

Lesson 05 / interface

interfaceは「この形で使える」という契約。

interfaceは、クラスに実装してほしい振る舞いを決めます。普通の抽象メソッドはpublic abstract扱いです。フィールドはpublic static final扱いです。

interface Printable {
  int SIZE = 10;
  void print();

  default void log() {
    System.out.println("log");
  }
}
contract

実装すべき形を決める

implementsしたクラスは、必要なメソッドを実装します。

default

defaultは本体を持てる

interface内でも、default methodは処理を書けます。

constant

変数は定数扱い

interfaceのフィールドはpublic static finalとして扱われます。

Tracea

勘違い注意: interfaceの普通のメソッドはpublicです。

実装クラス側でアクセスを狭くすると、オーバーライド条件を満たせません。Silverでは「publicを付け忘れている実装」が罠になります。

前へ次のLessonへ

Lesson 06 / overload vs override

オーバーロードは「別メニュー」、オーバーライドは「上書き」。

名前が似ていますが、試験ではまったく違う判定をします。オーバーロードは同じ名前で引数が違うメソッドを増やすこと。オーバーライドは親のメソッドを子で置き換えることです。

void print(String text) {}
void print(int number) {} // overload

class Child extends Parent {
  @Override
  void show() {} // override
}
overload

引数リストで区別

戻り値だけ違ってもオーバーロードにはなりません。

override

親と同じシグネチャ

アクセスを狭くできず、戻り値にもルールがあります。

@Override

ミス検出に役立つ

上書きできていなければコンパイル時に気づけます。

Tracea

勘違い注意: 戻り値だけを変えてもオーバーロードではありません。

Javaは呼び出し時の引数でどのメソッドか選びます。戻り値だけ違う同名同引数メソッドは区別できないので、コンパイルエラーです。

前へ次のLessonへ

Lesson 07 / immutable

immutableは「作った後に状態が変わらない」こと。

Stringは代表的なimmutableです。replaceやconcatを呼んでも、元のStringが変わるのではなく、新しいStringが返ります。問題では、戻り値を代入しているかを必ず見ます。

String s = "Java";
s.concat(" Silver");
System.out.println(s); // Java

s = s.concat(" Silver");
System.out.println(s); // Java Silver
String

Stringは変わらない

メソッドは新しい文字列を返します。

assign

代入して初めて変数が変わる

戻り値を受け取らないと、結果は捨てられます。

safe

安全に共有しやすい

勝手に中身が変わらないので、読みやすさと安全性が上がります。

Tracea

勘違い注意: finalとimmutableは同じではありません。

finalは変数の再代入を止める印です。immutableはオブジェクトの状態が変わらない性質です。言葉が近いだけで、見る場所が違います。

前へ次のLessonへ

Lesson 08 / record class

recordは「データを持つ小さな型」を短く書く仕組み。

recordは、値を運ぶためのクラスを簡潔に書く機能です。コンポーネントから、private finalフィールド、アクセサ、コンストラクタ、equals、hashCode、toStringが作られます。

record Student(String name, int score) {
}

Student s = new Student("Aoi", 90);
System.out.println(s.name());
System.out.println(s.score());
component

丸かっこ内がデータ

String nameやint scoreをコンポーネントと呼びます。

accessor

アクセサはname()

getName()ではなく、コンポーネント名と同じメソッドです。

final

recordは継承されない

recordは暗黙的にfinalです。

Tracea

勘違い注意: recordは完全な深いimmutableを保証するわけではありません。

recordのフィールドは再代入できません。ただし、コンポーネントに可変オブジェクトを渡した場合、その中身まで自動で守られるわけではありません。
