Silver Focus 02

継承は「参照型」と「実体型」を分けて読む。

Silverの継承問題は、親子関係を知っているだけでは足りません。変数の型で呼べるメソッドが決まり、実行時の実体でオーバーライド先が決まります。この2段階を分けます。

class Animal {
  void speak() {
    System.out.println("...");
  }
}

class Dog extends Animal {
  @Override
  void speak() {
    System.out.println("wan");
  }
}

Animal a = new Dog();
a.speak();
extends

子は親のメンバーを受け継ぐ

Dog extends Animal は、DogがAnimalの性質を持つという宣言です。ただし、コンストラクタは継承されません。

override

同じ形で上書きする

オーバーライドは、親のメソッドと同じシグネチャで振る舞いを差し替えます。戻り値、アクセス修飾子、例外の条件に注意します。

dynamic

実行時は実体のメソッドが動く

Animal a = new Dog(); でも、speak()はDog側が実行されます。ここが出力問題の核心です。

super

superは親側を見る合図

super() は親コンストラクタ、super.name や super.method() は親のメンバーを指します。

constructor

親のコンストラクタが先に動く

子を作ると、まず親の初期化が走ります。出力順問題では、コンストラクタ内のprintlnを上から追いましょう。

compile

呼べるかは左側の型で決まる

実体がDogでも、変数の型がAnimalならAnimalにないメソッドはそのまま呼べません。コンパイル可否を先に判定します。

Tracea Note

「呼べるか」と「何が動くか」を混ぜない。

左側の型は、コンパイル時に見える範囲を決めます。右側の実体は、オーバーライドされたメソッドの実行先を決めます。問題を読んだ瞬間に、左と右を別々に線で結んでください。

継承は便利な仕組みですが、試験では「親にない子だけのメソッド」「staticメソッドの隠蔽」「コンストラクタの順序」がよく混ざります。まずコンパイルできるか。次に出力順。焦るほど、この順番です。

参考: Oracle Java SE 17 Silver試験チェックリスト、Oracle Java Tutorials の Inheritance、 Overriding and Hiding Methods
